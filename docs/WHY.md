# WHY — feature rationale & promote log

> Agent entry point: [`AGENTS.md`](../AGENTS.md).

**Practice (adopted 2026-07-17):** every feature gets a line here answering *why it exists* — the real user pain or decision that drove it, not just what it does. This is the memory of the product's reasoning over time. When a feature ships, mark it **PROMOTE?** so we remember to announce/market the ones worth telling people about. Board tasks link here; this is the durable record.

Format: `FEATURE — WHY (the pain/decision) — [status] — PROMOTE: yes/no/later`

## Reliability / the rewrite
- **Go single-binary rewrite** — npm/node-gyp install friction + node-pty compile-on-install were the scariest, most failure-prone onboarding step; a static binary deletes it entirely. Also: Linux/VM future, MB-scale runners. — shipped — PROMOTE: **yes** (the trust + "no install scripts" story)
- **Lane ledger (recover/adopt)** — the fleet was lost to a mass-wipe THREE times; recovery was manual archaeology. Now it's one command. — shipped — PROMOTE: **yes** (never-lose-your-sessions is a headline)
- **Collision guard** — user hit it live: reopening a moved session "took over the line" (two drivers on one conversation). — shipped — PROMOTE: later
- **Codex app-server contract** — driving Codex by scraping its TUI is structurally fragile (stuck sends, prompt hijacks); the app-server gives a real JSON-RPC contract + attachable GUI. Reliability becomes structural, not scraped. — shipped — PROMOTE: **yes**
- **Claude `-p` structured path** — same idea for Claude, subscription-billed, side-by-side with the watchable PTY path. — shipped — PROMOTE: **yes**

## Orchestration / daily-use

- **CLI fixes (`--` passthrough, complete `--help`, `run --wait`)** — real dogfood bugs: child commands lost their flags, half the commands were undiscoverable. — shipped — PROMOTE: no (table stakes)
- **Self-documenting CLI; no required agent skill** — a coding agent should not need a separately installed skill to discover Sessions. `sessions help` lists the complete command surface, `sessions help <command>` gives focused usage and examples, and `sessions docs` renders the full offline Markdown reference from that same executable registry. The checked-in CLI reference is generated by the command rather than maintained separately. — shipped in source 2026-07-21 — PROMOTE: **yes**
- **`sessions search`** — you can't find a decision once it scrolls off screen, even though it is still on disk. The first version proved the data was searchable, but the PM-lane dogfood on 2026-07-23 exposed the actual job: “reconstruct every direction I gave about this topic, across the Claude manager and Codex worker, in order, and tell me what happened after each request.” Literal phrase guessing and session-grouped snippets could not do that. Search is therefore a retrieval workflow, not a grep box: ranked broad-token recall is the default; exact and regex remain explicit; results carry relevance, provider/machine/workspace/ancestry metadata, a stable message index, and optional neighboring turns; role, multiple-session, lane-name, workspace, provider, and date filters are composable; timeline mode merges matching moments chronologically. — transcript-aware recall implemented in source 2026-07-23 — PROMOTE: **yes**
- **Searchable user intent and relay history** — the PM reconstruction failed partly because scheduled babysit ticks and child-completion notifications appeared as “User,” while the actual Claude `Agent`, Codex `spawn_agent`, `send_message`, and `followup_task` bodies that moved work between managers and workers were discarded. Synthetic prompts are now normalized as automation, actual child creation as delegation, cross-lane messages as handoffs, and waits/reads as status; arbitrary shell/tool payloads remain excluded. This makes “Your requests” a trustworthy default and makes the Operations filter describe the cross-session bus honestly without indexing general command output or secrets. — implemented in source 2026-07-23 — PROMOTE: **yes**
- **Message-anchored transcript reader** — a hit is only useful if it lands at the remembered moment. Each result now opens the exact full-history message read-only using a content-derived bookmark, returning 409 instead of highlighting the wrong ordinal after a rewrite. Around, subsequent, request-only, full, and request-to-request views are streamed and rendered in pages of at most 500 original message positions; Back returns to the preserved query and filters, and viewing never resumes or sends. Search results omit full message bodies, deleted histories are purged from the plaintext local index, source fingerprints use the actual file rather than unrelated runner activity, parsing stops on cancellation, and non-AI keyword modes search only after explicit submission instead of rescanning a changing 387 MB source for every keystroke. — implemented in source 2026-07-23 — PROMOTE: **yes**
- **Explicit AI search planner** — people often remember the idea of a conversation but phrase the request as “the session where I talked about X.” Native Smart mode makes exactly one tool-disabled call through the user's already-authenticated Codex (default) or Claude CLI to remove that framing and produce a recall-oriented FTS5 query. It sends only the search request, never transcripts, snippets, session IDs, results, or the index. The generated query still runs against the local SQLite index, while role/provider/date/session filters remain deterministic. Keyword, exact, and regex search remain available without model calls. Planner calls are serialized; identical provider/query requests use hashed, bounded ten-minute cache entries so impatient clicks cannot fan out account usage or retain raw natural-language keys. — implemented in source 2026-07-22; recall prompt corrected 2026-07-23 — PROMOTE: **yes**
- **Lane `--description` (+ first-message fallback)** — at cleanup you can't tell WHY a lane exists → hard to know what's safe to kill. Give each lane a purpose. — shipped — PROMOTE: later
- **Mini dogfood cleanup: truthful doctor/version/file deltas plus safe retention** — the first 19-lane migration proved the compatibility path, then exposed four trust-eroding details: `doctor` could SIGHUP its own PTY probe, packaged binaries still announced 0.1.0, closed rows had no way to recede, and a lane inherited the whole repository's dirty count as if it wrote those files. Sessions.app also updates every existing managed CLI link into a standard command directory when safe, never replacing another tool. Source: Mini feedback `d1e0599d-8a51-489c-b50c-5e0142f7b97b`; canonical Somewhere task `tsk_c315cf9ed402416d8c6ba5f16072bef8`. — implemented in source 2026-07-23; release/notarization pending — PROMOTE: no (table stakes)

## Usage and organization (user-promoted 2026-07-19)
- **Usage analytics + arbitrary session tags** — token and cost totals are only marginally useful if users cannot connect them to the way they actually organize work: product line, client, team, project, cost center, experiment, or their own dimensions. Tags are now first-class daemon-owned key/value metadata, and the local SQLite usage index incrementally normalizes Claude and Codex logs into daily/weekly/monthly/session/tag/provider/model reports plus a polished dashboard with durable local saved views and expandable token/cost drill-down. Structured sessions also write usage live at the existing manager event boundary; stable provider/turn or provider/message keys make the later log backfill enrich the live row rather than count it twice. Users may explicitly save local default tags for recurring contexts; new sessions inherit them visibly and keep them editable before start, so Sessions never guesses or overwrites an explicit tag. Reasoning is surfaced as an explicit subset of output tokens instead of being double-counted. The schema-versioned API follows `ccusage`'s proven auto/display/calculate cost semantics and pins pricing provenance, while preserving Sessions' static Go runtime and avoiding an `npx` product dependency. Unknown model prices stay visibly unpriced instead of being guessed. Raw usage, filters, and tags stay local by default. — shipped 2026-07-20 — PROMOTE: **yes**
- **Fork snapshots are context, not new usage** — Codex child-agent rollouts carry a timestamp-rewritten copy of the parent task, including its historical `task_started` records. Indexing cross-checks those turn identities against the parent conversation: copied parent turns are never re-dated or re-billed, and equal usage from a different physical log cannot replace original provenance. Only turn identities unique to the child belong to the child conversation. — corrected 2026-07-22 — PROMOTE: **yes**
- **Opt-in daily work journal** — raw totals and a lane list answer "how much" and "what ran," but not the human question "what did I actually accomplish today?" Today keeps the authoritative timeline, hierarchy, tags, projects, outcomes, and usage local and useful with zero model calls. Written synthesis is native-app-only and `off` by default; when requested it makes one cached call through the user's already-authenticated Codex (recommended) or Claude CLI. Sessions never hardcodes a model: the selected CLI uses its own default, while Sessions explicitly requests its lowest supported reasoning effort and stores only the final answer. Codex runs ephemeral/read-only with user configuration and rules ignored after feature preflight; Claude uses safe mode with customizations, Chrome, slash commands, settings sources, tools, MCP, and session persistence disabled. The prompt arrives on stdin in an isolated temporary directory, strips provider API-key environment variables so it stays on the existing CLI account, and is hard-capped at 32 KiB of compact metadata and final structured summaries rather than transcripts or durable session IDs. Only the resulting Markdown is stored on this Mac. No Sessions API key or token markup exists (`runtime/internal/recap/service.go`, `frontend/src/components/TodayView.tsx`). — implemented in source 2026-07-22 — PROMOTE: **yes**
- **Today includes work that is still outside Sessions** — local usage already observes Claude and Codex provider logs, so a factual journal that reports tokens but says no work occurred is internally inconsistent. Today now labels both managed Sessions lanes and local outside conversations. It streams only the physical logs that contributed usage in the selected day, keeps bounded message excerpts, excludes Codex fork snapshots, and imports no transcript. — implemented in source 2026-07-22 — PROMOTE: **yes**
- **Adopt, do not transfer** — a provider conversation has one durable identity and source history. Bringing it into Sessions is a deliberate recovery/adopt operation after the original client stops writing. The picker never copies transcript files, hides identities already open in Sessions, and relies on existing live/moved/collision guards; resuming from two clients at once is explicitly unsafe. This avoids both transfer loss and duplicate histories. — implemented in source 2026-07-22 — PROMOTE: **yes**

## Remote security (codex audit 2026-07-18 — decision record)
Verdict: **keep Tailscale Serve as the only remote transport.** It's the best primitive (private reachability + E2E-encrypted transport + no Sessions data plane + tiny code), not the only one. Full analysis in scratchpad `remote-security-verdict.md`.
- **The real trust gap is mutable browser JS, not the network.** "No Sessions data plane" is true; "cryptographically impossible for us to see data" is NOT while browser code can drive the daemon. The 2026-07-22 decision therefore retires interactive browser control in favor of signed native clients. If a read-only status page remains, daemon-served same-origin assets are still preferable to mutable hosted JS.
- **Origin allowlist is NOT a second factor** — it only limits browser pages; doesn't stop curl or a leaked token. Don't market it as auth.
- **`/api/health/deep` is unauthenticated** (leaks session IDs/PIDs/dims). Tailnet-only today = not urgent; MUST authenticate before any public ingress. — deferred (not chasing now)
- **Token-hardening ladder (do BEFORE adding any new ingress):** one-time pairing tickets → per-device keys → protected native credential storage + scopes → short-lived session tokens → `sessions devices list/revoke` → no permanent token in WS URLs. — RUNG 1 SHIPPED 2026-07-19: **`sessions pair` QR one-time pairing** (single-use 5-min tickets, per-device tokens stored hashed, rate-limited claim, `sessions devices`/`revoke`). NATIVE UX REPLACED 2026-07-23: on Tailscale, the client discovers healthy Sessions peers, sends a named request, and waits while the host sees the Tailscale account identity and explicitly accepts or denies. Tailscale Serve identity is trusted only from the immediate loopback proxy; public bootstrap routes categorically reject browser Origins and non-JSON requests; a healthy peer grants no API access; and the normal revocable device token exists only after the accepted requester claims it. Accepted claims are idempotent, while the token stays time-bounded/pending until its first authenticated API use, so a lost 201 cannot leave an orphaned live credential. The displayed device name is explicitly self-reported because Serve proves the tailnet user, not the node; a malicious local process can fabricate the display identity but already holds loopback daemon authority. QR/link pairing is now only the private-LAN fallback for devices without Tailscale. Both flows key the saved entry by stable daemon identity—no SSH, duplicate LAN/Tailscale machines, or master-token copying. The remaining hardening is explicit rather than implied: protect native credentials at rest, add client scopes, then issue short-lived WebSocket tickets. — PROMOTE: **yes** (recognizable-account approval, never a shared master secret)
- **Workspace discovery must not cause privacy prompts — implemented 2026-07-23** — the New Session launcher no longer sweeps the whole home directory or stats Desktop, Documents, and Downloads just to suggest folders. It expands conventional local development roots, puts project candidates first, and warns when the user explicitly chooses the whole home folder. A runner still inherits the chosen workspace: an agent launched in `/Users/<name>` can legitimately encounter Music or cloud-drive TCC boundaries, so broad workspace choice remains visible rather than being silently sandboxed. — PROMOTE: **yes** (honest permissions)
- **Don't build a custom relay; don't default Funnel.** If phone-VPN friction proves real, add opt-in Tailscale Funnel AFTER the token ladder — never on by default. — deferred
- **Semantic search (LOCAL embeddings) — ONLY IF FTS proves insufficient in real use** — find by meaning with ZERO shared vocabulary. If earned: LOCAL embedding model only (cloud embed API breaks "we never see your data"), opt-in, incremental, off critical path. until Somewhere exposes the Sessions-specific account and provisioning APIs.

Cloud usage is a separate metadata stream and does not require plaintext
transcripts. Password recovery will wrap the random archive key using a
memory-hard password-derived key plus authenticated encryption. Bcrypt remains
appropriate for one-way password verification, not recoverable session
encryption.

# 2026-07-22 — Make the native app an agent operations inbox

The product center is the five manager conversations and the health of their
children, not a terminal multiplexer. The native shell therefore keeps a small
global rail, a lineage-aware session navigator, and one large active-session
area. Only sessions explicitly opened from the navigator become tabs. Closing a
tab is view-only; ending a process is a separate dangerous action in Details.
Terminal remains one click away as an escape hatch rather than defining the
information architecture.

Hierarchy must use the daemon's append-only creator ledger. The frontend
normalizes the existing snake-case provenance fields but never guesses a parent
from time, cwd, or title. The inbox asks for exited sessions so a completed
parent does not disappear and promote its live child into a false manager. Up to
five manager pins are a bounded local UI
preference and do not change provenance. Delegation uses the trusted creator
header, inherits profile context only while the provider matches, and sends at
most one initial task after the provider is ready using a bracketed paste and a
separate Enter PTY write; Sessions still has no secret prompt queue.
New isolated provider homes are the exception: login must complete first, so the
launcher never applies wait-ready or task input to that authentication flow.

Exited sessions belong in the lineage navigator but not in live transports.
Opening one uses the bounded read-only history preview plus Details; it never
attaches xterm or WebSocket. Grid and mobile quick switching remain live-only,
and a finished intermediate node cannot collapse while it is the path to a live
descendant.

Global creation starts with the optional request and recent workspaces instead
of a filesystem browser. Profile, worktree, model, and permissions are advanced
concerns. Current provider-profile and worktree controls remain real; explicit
model selection and normalized inline child start/completion cards are Coming
soon. Both dark and light surfaces follow the supplied macOS mockups.

# 2026-07-22 — Treat local agent discovery and app startup as product paths

The native app launches runners through launchd, not through a login shell, so
its PATH must explicitly include the conventional user CLI locations where
Claude, Codex, Cargo, Bun, and npm-installed tools live. A missing provider
binary is knowable before a runner is launched; Sessions now reports that
directly instead of making the user wait sixty seconds for a socket that can
never exist.

Installed runtime directories are immutable. A generic Git `-dirty` suffix
therefore cannot identify local builds safely; dirty source packages carry a
stable content fingerprint. During an update, port release is necessary but
not sufficient: launchd must also finish removing the old service before a new
bootstrap. Both rules preserve the "one app update, no lost sessions" contract
and were proven with the scratch launchd test plus a live Claude runner carried
through the final packaged update.

New Session discovery is a convenience, never a prerequisite. macOS may block
while hydrating iCloud-backed Desktop, Documents, or Downloads directories, so
the daemon expands only the home folder and conventional local development
roots; the launcher renders useful fallback workspaces immediately and keeps
the explicit folder picker available.

Usage indexing should feel like app state, not a report the user has to start.
The app warms the default local report after daemon connection and shares that
request/cache with the Usage surface. It remains local, non-blocking, and
errors stay owned by the dashboard rather than breaking the sessions inbox.

# 2026-07-22 — Make provider defaults typed, inherited, and daemon-authoritative

Provider defaults must behave the same whether a session starts from the native
launcher, the CLI, delegation, or the future mini. Sessions therefore stores a
small typed Claude policy and resolves it inside `sessionsd`, with explicit
per-session choices taking precedence and `inherit` deferring to Claude. It
does not edit `~/.claude/settings.json`, and it does not offer a free-form
startup-command box.

Remote Control is a launch capability (`--remote-control`), not a secret prompt
that Sessions should paste after startup. Permission mode, model, effort,
Chrome, and Remote Control naming likewise become explicit arguments only when
selected. The historical bypass-permissions posture remains the compatibility
default but is now visible and can be replaced globally or for one launch.

Somewhere MCP uses the same by-reference authentication rule as backup. The
preferred managed definition launches `somewhere mcp`; that adapter reads the
Somewhere CLI credential itself. Sessions recognizes the canonical HTTP form
and stdio adapter under any name, does not add a duplicate, and refuses to
overwrite a different server named `somewhere`. Token bytes never enter a
prompt, runner environment, temporary MCP file, or Sessions settings.

# 2026-07-23 — Treat agents as first-class support reporters

Sessions is agent-native: a substantial share of failures and product feedback
will be encountered by Codex, Claude, or shell automation before a person sees
them. A Settings-only intake would therefore hide the support contract from the
product's primary operators.

The CLI JSON support envelope is the agent surface. It names the stable
machine-readable command, the small safe set of report fields, and the
requirement to ask the user before opening or submitting a ticket. Agents report
the sanitized shape of the failing Sessions command or app action, exit code,
sanitized exact error, expected/actual behavior, and repeatability; they do not
dump transcripts, terminal output, paths, identifiers, environment,
credentials, private source, logs, or crash files. Public intake records
whether the report originated with an agent, direct use, or both. Agent-native
reporting changes discoverability and structure, not consent: Sessions still
performs no automatic upload or submission.

# 2026-07-23 — Make Fleet a device picker, not an infrastructure diagram

People should not need identical Sessions releases on every computer. Fleet
compares the daemon versions it already receives from health and explains when
another machine is older, newer, or simply different, but version skew alone
does not disable it. A future hard compatibility decision must come from an
explicit daemon protocol range, not an inferred semantic-version comparison.

Adding a machine belongs in Fleet, where users expect to find devices.
`Find machines` therefore reuses the shipped tailnet candidate scan, Sessions
health verification, and named host approval flow. This follows the useful
part of consumer device discovery: find candidates automatically, then verify
the expected service before showing or trusting them. Nearby-Wi-Fi
Bonjour/mDNS is a separate coming-soon transport because adding a service
advertisement and macOS Local Network permission deserves an explicit product
decision rather than an incidental prompt.

Visual hierarchy communicates state before labels do. The current computer is
the bright, primary card; an unreachable machine fades as a whole card; and
the Somewhere VM is muted because it is an unprovisioned placeholder, not an
offline machine the user can repair. Readable operational text uses normal
foreground contrast instead of making every secondary fact look disabled.

# 2026-07-23 — Ship Windows first as a paired native client

Windows does not need to host agents to be immediately useful. Its first
release is the same native operations inbox pointed at Mac-hosted Sessions
machines: discover verified candidates through the user's existing Tailscale
client, request access, require approval on the host, and store only the normal
revocable device credential. Manual endpoint entry remains an escape hatch.

The shared frontend must not create `localhost:8787` on a client-only platform.
Only macOS owns the launchd lifecycle and bundled Go runtime for now; Windows,
Android, and iOS begin as clients. This keeps the runner boundary small without
turning the deprecated browser into the remote-control product.

Windows installers are built on a real Windows CI runner because packaging and
WebView behavior should be exercised on the target OS. CI may provide an
unsigned, short-retention NSIS preview for private testing, but a public
download requires Authenticode plus the existing signed-updater discipline.
SmartScreen reputation is a distribution concern to solve, not a warning to
teach users to ignore.
