# WHY — feature rationale & promote log

> Agent entry point: [`AGENTS.md`](../AGENTS.md).

**Practice (adopted 2026-07-17):** every feature gets a line here answering *why it exists* — the real user pain or decision that drove it, not just what it does. This is the memory of the product's reasoning over time. When a feature ships, mark it **PROMOTE?** so we remember to announce/market the ones worth telling people about. Board tasks link here; this is the durable record.

Format: `FEATURE — WHY (the pain/decision) — [status] — PROMOTE: yes/no/later`

## Reliability / the rewrite
- **Go single-binary rewrite** — npm/node-gyp install friction + node-pty compile-on-install were the scariest, most failure-prone onboarding step; a static binary deletes it entirely. Also: Linux/VM future, MB-scale runners. — shipped — PROMOTE: **yes** (the trust + "no install scripts" story)
- **Lane ledger (recover/adopt)** — the fleet was lost to a mass-wipe THREE times; recovery was manual archaeology. Now it's one command. — shipped — PROMOTE: **yes** (never-lose-your-sessions is a headline)
- **Collision guard** — user hit it live: reopening a moved session "took over the line" (two drivers on one conversation). — shipped — PROMOTE: later
- **Codex app-server contract** — driving Codex by scraping its TUI is structurally fragile (stuck sends, prompt hijacks); the app-server gives a real JSON-RPC contract + attachable GUI. Reliability becomes structural, not scraped. — shipped — PROMOTE: **yes**
- **Claude `-p` structured path** — same idea for Claude, subscription-billed, side-by-side with the watchable PTY path. — shipped — PROMOTE: **yes**

The checked-in CLI reference is generated by the command rather than maintained separately. — shipped in source 2026-07-21 — PROMOTE: **yes**
- **`sessions search`** — you can't find text in a chat once it scrolls off screen, though it's all on disk. Search should be a command. — shipped (exact substring/regex) — PROMOTE: **yes** (obvious, universally-wanted)

Search state survives navigation, result clicks use a distinct old-runtime-safe endpoint for a bounded read-only tail preview, and resuming remains a separate action. ## Usage and organization (user-promoted 2026-07-19)
- **Usage analytics + arbitrary session tags** — token and cost totals are only marginally useful if users cannot connect them to the way they actually organize work: product line, client, team, project, cost center, experiment, or their own dimensions. Tags are now first-class daemon-owned key/value metadata, and the local SQLite usage index incrementally normalizes Claude and Codex logs into daily/weekly/monthly/session/tag/provider/model reports plus a polished dashboard with durable local saved views and expandable token/cost drill-down. Structured sessions also write usage live at the existing manager event boundary; stable provider/turn or provider/message keys make the later log backfill enrich the live row rather than count it twice. Users may explicitly save local default tags for recurring contexts; new sessions inherit them visibly and keep them editable before start, so Sessions never guesses or overwrites an explicit tag. Reasoning is surfaced as an explicit subset of output tokens instead of being double-counted. The schema-versioned API follows `ccusage`'s proven auto/display/calculate cost semantics and pins pricing provenance, while preserving Sessions' static Go runtime and avoiding an `npx` product dependency. Unknown model prices stay visibly unpriced instead of being guessed. Raw usage, filters, and tags stay local by default. — shipped 2026-07-20 — PROMOTE: **yes**
- **Fork snapshots are context, not new usage** — Codex child-agent rollouts carry a timestamp-rewritten copy of the parent task, including its historical `task_started` records. Indexing cross-checks those turn identities against the parent conversation: copied parent turns are never re-dated or re-billed, and equal usage from a different physical log cannot replace original provenance. Only turn identities unique to the child belong to the child conversation. — corrected 2026-07-22 — PROMOTE: **yes**
- **Opt-in daily work journal** — raw totals and a lane list answer "how much" and "what ran," but not the human question "what did I actually accomplish today?" Today keeps the authoritative timeline, hierarchy, tags, projects, outcomes, and usage local and useful with zero model calls. Written synthesis is native-app-only and `off` by default; when requested it makes one cached call through the user's already-authenticated Codex (recommended) or Claude CLI. Sessions never hardcodes a model: the selected CLI uses its own default, while Sessions explicitly requests its lowest supported reasoning effort and stores only the final answer. Codex runs ephemeral/read-only with user configuration and rules ignored after feature preflight; Claude uses safe mode with customizations, Chrome, slash commands, settings sources, tools, MCP, and session persistence disabled. The prompt arrives on stdin in an isolated temporary directory, strips provider API-key environment variables so it stays on the existing CLI account, and is hard-capped at 32 KiB of compact metadata and final structured summaries rather than transcripts or durable session IDs. Only the resulting Markdown is stored on this Mac. No Sessions API key or token markup exists (`runtime/internal/recap/service.go`, `frontend/src/components/TodayView.tsx`). — implemented in source 2026-07-22 — PROMOTE: **yes**
- **Today includes work that is still outside Sessions** — local usage already observes Claude and Codex provider logs, so a factual journal that reports tokens but says no work occurred is internally inconsistent. Today now labels both managed Sessions lanes and local outside conversations. It streams only the physical logs that contributed usage in the selected day, keeps bounded message excerpts, excludes Codex fork snapshots, and imports no transcript. — implemented in source 2026-07-22 — PROMOTE: **yes**
- **Adopt, do not transfer** — a provider conversation has one durable identity and source history. Bringing it into Sessions is a deliberate recovery/adopt operation after the original client stops writing. The picker never copies transcript files, hides identities already open in Sessions, and relies on existing live/moved/collision guards; resuming from two clients at once is explicitly unsafe. This avoids both transfer loss and duplicate histories. — implemented in source 2026-07-22 — PROMOTE: **yes**

# 2026-07-22 — Show the cloud product shape before claiming the cloud control plane exists

The native UI may operate the backup feature that already exists: the bundled
Sessions CLI reads the local Somewhere CLI identity by reference, configures an
existing user-owned project, encrypts locally with XChaCha20-Poly1305, and pushes
directly. The UI must label the account library, hosted search, central cloud

Cloud usage is a separate metadata stream and does not require plaintext

appropriate for one-way password verification, not recoverable session

# 2026-07-22 — Make the native app an agent operations inbox

The product center is the five manager conversations and the health of their
children, not a terminal multiplexer. The native shell therefore keeps a small
global rail, a lineage-aware session navigator, and one large active-session
area. Only sessions explicitly opened from the navigator become tabs. Closing a
tab is view-only; ending a process is a separate dangerous action in Details.
Terminal remains one click away as an escape hatch rather than defining the
information architecture.

Hierarchy must use the daemon's append-only creator ledger. The frontend
normalizes the existing snake-case provenance fields but never guesses a parent
from time, cwd, or title. The inbox asks for exited sessions so a completed
parent does not disappear and promote its live child into a false manager. Up to
five manager pins are a bounded local UI
preference and do not change provenance. Delegation uses the trusted creator
header, inherits profile context only while the provider matches, and sends at
most one initial task after the provider is ready using a bracketed paste and a
separate Enter PTY write; Sessions still has no secret prompt queue.
New isolated provider homes are the exception: login must complete first, so the
launcher never applies wait-ready or task input to that authentication flow.

Exited sessions belong in the lineage navigator but not in live transports.
Opening one uses the bounded read-only history preview plus Details; it never
attaches xterm or WebSocket. Grid and mobile quick switching remain live-only,
and a finished intermediate node cannot collapse while it is the path to a live
descendant.

Global creation starts with the optional request and recent workspaces instead
of a filesystem browser. Profile, worktree, model, and permissions are advanced
concerns. Current provider-profile and worktree controls remain real; explicit
model selection and normalized inline child start/completion cards are Coming
soon. Both dark and light surfaces follow the supplied macOS mockups.

# 2026-07-22 — Treat local agent discovery and app startup as product paths

The native app launches runners through launchd, not through a login shell, so
its PATH must explicitly include the conventional user CLI locations where
Claude, Codex, Cargo, Bun, and npm-installed tools live. A missing provider
binary is knowable before a runner is launched; Sessions now reports that
directly instead of making the user wait sixty seconds for a socket that can
never exist.

Installed runtime directories are immutable. A generic Git `-dirty` suffix
therefore cannot identify local builds safely; dirty source packages carry a
stable content fingerprint. During an update, port release is necessary but
not sufficient: launchd must also finish removing the old service before a new
bootstrap. Both rules preserve the "one app update, no lost sessions" contract
and were proven with the scratch launchd test plus a live Claude runner carried
through the final packaged update.

New Session discovery is a convenience, never a prerequisite. macOS may block
while hydrating iCloud-backed Desktop, Documents, or Downloads directories, so
the daemon expands only the home folder and conventional local development
roots; the launcher renders useful fallback workspaces immediately and keeps
the explicit folder picker available.

Usage indexing should feel like app state, not a report the user has to start.
The app warms the default local report after daemon connection and shares that
request/cache with the Usage surface. It remains local, non-blocking, and
errors stay owned by the dashboard rather than breaking the sessions inbox.

# 2026-07-22 — Make provider defaults typed, inherited, and daemon-authoritative

Provider defaults must behave the same whether a session starts from the native
launcher, the CLI, delegation, or the future mini. Sessions therefore stores a
small typed Claude policy and resolves it inside `sessionsd`, with explicit
per-session choices taking precedence and `inherit` deferring to Claude. It
does not edit `~/.claude/settings.json`, and it does not offer a free-form
startup-command box.

Remote Control is a launch capability (`--remote-control`), not a secret prompt
that Sessions should paste after startup. Permission mode, model, effort,
Chrome, and Remote Control naming likewise become explicit arguments only when
selected. The historical bypass-permissions posture remains the compatibility
default but is now visible and can be replaced globally or for one launch.

Somewhere MCP uses the same by-reference authentication rule as backup. The
preferred managed definition launches `somewhere mcp`; that adapter reads the
Somewhere CLI credential itself. Sessions recognizes the canonical HTTP form
and stdio adapter under any name, does not add a duplicate, and refuses to
overwrite a different server named `somewhere`. Token bytes never enter a
prompt, runner environment, temporary MCP file, or Sessions settings.
