Let's get you running.

Install the native Mac app, choose exactly how far it can reach, then pair each device with one scan. The core needs no Sessions account, and nothing here sends terminal traffic through us.

0
Before you start
a 10-second inventory

For this walkthrough you need an Apple Silicon Mac and Claude Code and/or Codex already installed — Sessions runs the agents you already have; it does not replace or rebill them. You do not need Node, npm, Xcode, or a native build toolchain.

1
Install Sessions
on your Mac
native Mac app Download Sessions 0.1.0 →
brew install --cask somewhere-tech/tap/sessions-app

Move Sessions.app to Applications and open it. The app verifies and installs its own signed background runtime. There is no Node or npm install, no native-module compile, and no script piped into your shell.

For headless machines and developers, the same release also includes checksummed archives. Keep sessions, sessionsd, and sessions-runner together:

brew install somewhere-tech/tap/sessions

The tap pins the immutable release URL and SHA-256. The advanced installation guide also has direct-download verification steps and Linux archive names.

sessions install

sessions install registers the Go daemon with launchd, starts it, checks its health, and prints the local URL. Open that URL. Session runners live outside the daemon, so closing a terminal or reloading the daemon does not kill their work.

Open the local URL →
Stuck on install? Keep all three binaries in the same directory and make sure it is on PATH. sessions doctor checks the daemon and session runners.
Installation troubleshooting
sessions: command not found

Check that Homebrew's bin directory or $HOME/.local/bin is on your PATH, then open a new shell.

The daemon is down

Run sessions doctor. It checks the daemon and session runners and tells you what needs attention. There is no native-module or Xcode build step to repair.

2
Choose the reach
local first · each wider listener is opt-in

Sessions starts on 127.0.0.1:8787. Widen that boundary only when you want another device to connect.

your Mac · zero setup

Stay on localhost

Nothing to enable. Open http://localhost:8787; no network configuration or account is involved.

same Wi-Fi · no Tailscale

Open the LAN rung

Run sessions lan enable. Sessions binds a second listener to the current LAN address; protected routes still require a credential.

From anywhere: install Tailscale and sign in on the Mac and client device — tailscale.com/download — then run:

sessions remote enable

Sessions configures a tailnet-only Tailscale Serve HTTPS endpoint and verifies it. TLS terminates on your Mac; there is no Sessions relay and no open internet port.

One honest heads-up: Tailscale's HTTPS certificate lists your machine + tailnet name in public certificate-transparency logs. Harmless for most, but if your machine name is sensitive, rename it before you enable serve.
Remote-access troubleshooting
The .ts.net address does not resolve

Let this device use Tailscale DNS with tailscale set --accept-dns=true. Then confirm MagicDNS is enabled in the Tailscale DNS console and retry sessions remote status.

The address resolves but does not answer

Make sure both devices are signed into the same tailnet, then run sessions doctor and sessions remote status on your Mac.

3
Pair the device
scan once · zero typing · no shared master secret

After enabling LAN or remote access, mint a one-time QR for the device in your hand:

sessions pair --name "my phone"

The ticket expires after five minutes and works once. It is exchanged directly with your Mac for a per-device credential; the daemon's master token is never embedded in the QR.

sessions devices
sessions devices revoke <id-or-prefix>

List each credential by device and revoke one immediately without rotating every other device.

4
Back up the history
optional · encrypted before upload

If you have a somewhere project, Sessions can push its session history there directly. Turn on local XChaCha20-Poly1305 encryption so the transcripts and manifest leave your Mac only as ciphertext:

sessions backup enable --project my-project --interval 15m --encrypt

Write down the recovery phrase. The key stays on your machine, somewhere receives encrypted bytes, and there is no Sessions-operated backup relay. Without that phrase, nobody can reset or recover the key for you.

sessions backup status

Uninstall cleanly.

End any sessions you still need first. Then let sessions stop the daemon and remove its launchd registration before uninstalling the Homebrew package. Session and recovery data are left in place.

sessions uninstall
brew uninstall sessions
